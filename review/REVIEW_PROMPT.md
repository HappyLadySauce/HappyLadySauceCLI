# Code Review Prompt（代码审查提示词模板）

## 使用方法

将以下提示词发送给 AI 编码助手，替换 `{COMMIT_RANGE}` 为实际提交范围（如 `HEAD~3..HEAD`）。

---

## 提示词正文

```
请对 `{COMMIT_RANGE}` 的代码变更进行全面审计。按以下结构化流程执行：

## 审查流程

### 第一阶段：架构理解
1. 从程序入口（main / RunLoop / 初始化函数）开始，逐层追踪调用链
2. 画出完整调用图：入口 → 中间件链 → 业务逻辑 → 外部依赖
3. 标注每一层的数据流向和错误传播路径

### 第二阶段：逐函数追踪
对每个新增/修改的函数，按以下步骤分析：

**执行路径模拟：**
- 列出函数接收的所有可能输入组合（nil、零值、边界值、恶意值）
- 逐步追踪每种输入的执行分支，直到函数返回
- 标注每一步的副作用（I/O、状态变更、锁获取/释放）

**安全检查（对每个函数逐项打钩）：**
- [ ] 输入校验 —— 所有外部输入是否验证？长度/类型/范围是否限制？
- [ ] 错误处理 —— 错误是否被吞掉（`_` 丢弃）？错误信息是否泄露敏感数据？
- [ ] 资源管理 —— HTTP 请求/文件/锁是否设置了超时？defer 是否正确释放？
- [ ] 注入风险 —— 用户输入是否拼接到命令/SQL/URL 中？
- [ ] 认证授权 —— 敏感操作是否经过权限检查？是否有提权路径？
- [ ] 数据暴露 —— 日志/错误消息是否包含 API Key/Token/PII？

**性能检查（对每个函数逐项打钩）：**
- [ ] 锁粒度 —— 互斥锁持有期间是否执行了 I/O 或阻塞操作？
- [ ] 内存分配 —— 热路径上是否有不必要的 slice/map 分配？
- [ ] 重复计算 —— 同一数据是否被多次查询/计算？
- [ ] 超时设置 —— 所有 I/O 操作是否设置了合理的超时？
- [ ] 并发安全 —— 共享状态是否正确保护？有无 data race 风险？

### 第三阶段：跨层分析
- **中间件链顺序**：中间件的执行顺序是否合理？错误是否被正确传播？
- **数据一致性**：同一数据在不同层之间是否保持一致？
- **失败模式**：每个外部依赖失败时，系统是否优雅降级？

### 第四阶段：输出报告
按严重程度（CRITICAL > HIGH > MEDIUM > LOW）组织发现，每条包含：
- 严重度标记 + 简短标题
- 文件路径:行号（可点击跳转）
- 问题描述（1-2 句）
- 攻击场景/触发条件（安全/稳定性问题必须包含）
- 修复建议（代码片段优先）

## 输出格式
使用以下 Markdown 表格汇总所有发现：

| # | 严重度 | 文件:行号 | 类别 | 问题 | 修复建议 |
|---|--------|-----------|------|------|----------|
| 1 | 🔴 HIGH | `pkg/foo.go:42` | 安全 | ... | ... |

## 特别关注
- Go 语言特定：nil pointer dereference、goroutine leak、channel deadlock、slice 扩容
- 库使用：Eino ADK middleware 的正确实现、tool call 生命周期
- HTTP 客户端：超时、状态码检查、响应体大小限制
- 并发：锁的粒度、死锁风险、TOCTOU 竞态
```
